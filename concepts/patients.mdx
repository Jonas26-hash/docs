A patient record is the central object in the Patient Monitoring System. It holds demographic information, contact details, medical history, and insurance data for a single individual. Most clinical resources — appointments, prescriptions, and charges — reference a patient by their system-generated `id`, so creating accurate patient records before linking other data is important.

## Data model

The table below describes every field in a patient record.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | integer | Auto-generated | Unique numeric identifier assigned by the system. Do not supply this field when creating a patient. |
| `nombres` | string | Yes | Given name(s) of the patient. |
| `apellidoPaterno` | string | Yes | Paternal surname. |
| `apellidoMaterno` | string | No | Maternal surname. |
| `dni` | string | Yes | 8-digit national identity document number. Must be unique across all patient records. |
| `fechaNacimiento` | string | Yes | Date of birth in `YYYY-MM-DD` format (e.g., `"1990-03-15"`). |
| `genero` | string | No | Gender of the patient. |
| `direccion` | string | No | Residential address. |
| `telefono` | string | No | 9-digit phone number (e.g., `"987654321"`). |
| `email` | string | No | Contact email address. |
| `antecedentesFamiliares` | string | No | Relevant family medical history. |
| `alergias` | string | No | Known allergies. |
| `condiciones` | string | No | Existing chronic or diagnosed conditions. |
| `medicamentosActual` | string | No | Medications the patient is currently taking. |
| `nombreSeguro` | string | No | Name of the patient's health insurance provider. |
| `numeroPoliza` | string | No | Insurance policy number. |
| `vigenciaSeguro` | string | No | Insurance expiry date in `YYYY-MM-DD` format. |
| `activo` | boolean | Auto-set | `true` for active records; `false` for soft-deleted records. Defaults to `true` on creation. |

<Note>
The `id` field is auto-generated by the system when you create a patient. You should never include it in `POST` request bodies.
</Note>

## Soft deletion

Deleting a patient with `DELETE /pacientes/{id}` does **not** permanently remove the record from the database. Instead, the system sets `activo` to `false`, preserving the record for audit and historical reference. Soft-deleted patients will not appear in standard list responses but remain accessible by their `id` if needed.

<Warning>
Because patient data is never fully erased, treat all patient records as permanent. Ensure you have the appropriate consent and data governance policies in place before creating records.
</Warning>

## Looking up a patient by DNI

If you know a patient's national identity number but not their system `id`, use the DNI lookup endpoint:

```bash
GET /pacientes/dni/{dni}
```

For example:

```bash
curl http://localhost:8080/pacientes/dni/45678901 \
  -H "Authorization: Bearer <TOKEN>"
```

This returns the full patient object for the matching record. Because `dni` is unique, the response is always a single object rather than a list.

<Tip>
When registering a new user with the `PACIENTE` role, you can supply a `dni` in the registration request to automatically create and link a patient record. See [User Roles and Permissions](/concepts/roles) for details.
</Tip>
